## Энкодер

### Процесс кодирования

Кодирование выполняется конвейером из трёх последовательных преобразований:

#### 1. Преобразование Барроуза-Уилера (BWT)

Входной файл целиком подаётся на BWT. Строится массив циклических ротаций исходных данных, который затем сортируется
лексикографически. Для эффективной сортировки используется метод удвоения префиксов (prefix doubling) с учётом
циклических обёрток (`(i + k) % n`).

Результат BWT — последний столбец отсортированной матрицы ротаций и индекс строки, соответствующей исходным данным. BWT
группирует одинаковые символы рядом, создавая длинные серии повторов.

#### 2. Преобразование Move-to-Front (MTF)

Выход BWT подаётся на MTF. Поддерживается динамический список из 256 байтовых значений. Для каждого входного символа
выводится его текущая позиция в списке, после чего символ перемещается в начало списка.

Благодаря тому, что BWT сгруппировал повторяющиеся символы, MTF преобразует поток в последовательность с преобладанием
малых значений (особенно нулей, что хорошо работает с энтропийным кодированием).

#### 3. Каноническое кодирование Хаффмана

По выходу MTF строится частотная таблица побайтовых значений. На её основе строится дерево Хаффмана с использованием
приоритетной очереди (`PriorityQueue`) с детерминированным разрешением коллизий — при равных частотах приоритет
определяется порядком вставки, что гарантирует идентичное дерево при кодировании и декодировании.

Из дерева извлекаются длины кодов для каждого символа. Далее по длинам генерируются канонические коды Хаффмана — коды
определяются однозначно по паре (длина кода, значение символа), отсортированной лексикографически. Это позволяет хранить
в архиве только 256 байт длин кодов вместо полной частотной таблицы.

Выходной поток MTF кодируется побитно с использованием полученных канонических кодов (MSB-first).

### Формат архива

В выходной файл последовательно записываются:

1. Размер исходного файла (`int32`, 4 байта)
2. Индекс BWT (`int32`, 4 байта)
3. Таблица длин кодов (256 байт — по одному байту на каждый возможный символ)
4. Длина сжатых данных (`int32`, 4 байта)
5. Сжатые данные

Фиксированный overhead метаданных: **268 байт** на файл.

## Декодер

### Процесс декодирования

Декодирование выполняется в обратном порядке:

1. Из архива считываются метаданные: размер оригинала, индекс BWT, таблица длин кодов (256 байт), длина сжатых данных
2. По таблице длин восстанавливаются канонические коды Хаффмана и строится декодирующее дерево
3. Сжатые данные декодируются побитно обходом дерева Хаффмана до восстановления исходного количества символов
4. Декодированный поток подаётся на обратное MTF-преобразование (по индексу извлекается символ из динамического списка и
   перемещается в начало)
5. Результат MTF подаётся на обратное BWT-преобразование с использованием LF-mapping и сохранённого индекса,
   восстанавливая исходные данные
